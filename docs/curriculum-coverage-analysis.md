## Priority Build Queue

Based on exam weight and gap severity, here is the recommended build order:

### P0 — High Impact, High Feasibility
1. **`tool_choice` demo** (Domain 2.3) — Add examples showing `auto` / `any` / `{ type: "tool", name: "..." }` switching. Small code addition with high exam coverage.
2. **Built-in tool integration** (Domain 2.5) — Demonstrate Grep vs Glob and Read/Write/Edit patterns. Could be a utility module.
3. **Batch API skeleton** (Domain 4.4) — Even a stub showing `custom_id` correlation and sync-vs-batch routing would cover the exam concepts.

### P1 — Medium Impact
4. **Multi-instance review** (Domain 4.6) — Spawn independent review sessions; demonstrate per-file + cross-file passes.
5. **Task decomposition** (Domain 1.6) — Implement `decomposer.ts` on main with fixed pipeline and adaptive decomposition.
6. **Workflow enforcer** (Domain 1.4) — Merge `workflow-enforcer.ts` to main; it exists on other branches.
7. **Tool result trimming** (Domain 5.1) — Add a context-budget-aware trimmer that strips irrelevant fields from tool results.

### P2 — Lower Priority (config-based or niche)
8. **CI/CD integration script** (Domain 3.6) — Add a CI script using Claude Code with `-p` flag and JSON output.
9. **Provenance tracking** (Domain 5.6) — Add claim-source mapping to synthesis outputs.
10. **Access failure vs empty result** (Domain 5.3) — Extend `ToolError` to distinguish these two failure modes.

### Not Actionable in Code
- **Domain 3** (3.1–3.5) — These are configuration patterns, not code. Coverage comes from the `.claude/` directory structure, not `src/`. Consider adding a `docs/domain-3-walkthrough.md` explaining how the config maps to exam concepts.
